# Системные требования (System Requirements)

Требования к системе, описанные в этом документе, применимы **только к сервису приложения NocoBase** и охватывают вычислительные и ресурсы памяти, необходимые для процессов приложения. Они **не охватывают зависимые сторонние сервисы**, включая, помимо прочего:

- API-шлюзы / обратные прокси
- Сервисы баз данных (например, MySQL или PostgreSQL)
- Сервисы кэша (например, Redis)
- Промежуточное ПО, например очереди сообщений или объектное хранилище

За исключением проверки функциональности или полностью экспериментальных сценариев, **мы настоятельно рекомендуем развёртывать вышеупомянутые сторонние сервисы отдельно** на выделенных серверах или в контейнерах, либо используя соответствующие облачные сервисы.

Системную конфигурацию и планирование мощности этих сервисов необходимо оценивать и настраивать отдельно, исходя из **фактического объёма данных, нагрузки и уровня параллелизма**.

## Операционная система

Для производственных сред рекомендуется использовать Linux-серверы, например Ubuntu LTS, Debian, Rocky Linux или AlmaLinux. Docker, обратный прокси, управление процессами, логи и резервное копирование более зрелы в Linux и проще интегрируются с типичными облачными серверными средами.

NocoBase также можно установить на Windows Server. Если вы выбираете Windows Server, проверьте Node.js, базу данных, обратный прокси, супервизию процессов, резервное копирование, права на файлы, firewall и связанные операционные процессы в соответствии с фактическим способом развертывания.

macOS обычно больше подходит для локальной разработки и тестирования и не рекомендуется в качестве производственного сервера. Если вы настраиваете NocoBase локально для разработки или использования с ИИ-агентом, см. [Local Development Setup](../ai/local-development-setup.md).

## Одноузловой режим развертывания

Режим развертывания с одним узлом означает, что сервис приложения NocoBase работает на одном сервере или в одной контейнерной инстанции.

### Минимальные требования к оборудованию

| Ресурс | Требование |
| --- | --- |
| CPU | 1 ядро |
| Memory | 2 ГБ |

**Подходящие сценарии:**

- Микробизнес
- Доказательство концепции (POC)
- Среды разработки / тестирования
- Сценарии с почти отсутствующим одновременным доступом

::::info{title="Советы"}

- Эта спецификация гарантирует только то, что система сможет запуститься; она не определяет производительность.
- По мере роста объёма данных или числа одновременных запросов ресурсы системы могут быстро стать узким местом.
- Для **разработки исходного кода, разработки плагинов или сборки и развёртывания из исходников** выделите **как минимум 4 ГБ свободной памяти**, чтобы установка зависимостей, компиляция и шаги сборки выполнялись успешно.

::::

### Рекомендуемые требования к оборудованию

| Ресурс | Рекомендуемая спецификация |
| --- | --- |
| CPU | 2 ядра |
| Memory | ≥ 4 ГБ |

**Подходящие сценарии:**

Подходит для небольших и средних нагрузок с ограниченным параллелизмом в продакшн-средах.

::::info{title="Советы (Tips)"}

- С этой конфигурацией система справляется с типовыми административными операциями и лёгкими бизнес-нагрузками.
- При росте сложности бизнеса, одновременного доступа или фоновых задач рассмотрите увеличение спецификации оборудования или переход в кластерный режим.

::::

## Кластерный режим

Кластерный режим предназначен для средних и крупных нагрузок с более высоким уровнем параллелизма. Горизонтальное масштабирование позволяет повысить доступность и пропускную способность (см. [Кластерный режим](/cluster-mode) для подробностей).

### Требования к оборудованию для узлов

В кластерном режиме рекомендуемая аппаратная конфигурация для каждого узла приложения (Pod / инстанс) идентична режиму развертывания с одним узлом.

**Минимальная конфигурация для одного узла:**

- CPU: 1 ядро
- Memory: 2 ГБ

**Рекомендуемая конфигурация для одного узла:**

- CPU: 2 ядра
- Memory: 4 ГБ

### Планирование количества узлов

- Увеличивайте количество узлов по мере необходимости (2–N).
- Фактическое число узлов зависит от:
  - Одновременного трафика
  - Сложности бизнес-логики
  - Фоновых задач и асинхронных нагрузок
  - Времени реакции внешних зависимостей

Рекомендации для продакшн-сред:

- Динамически корректируйте количество узлов на основе показателей мониторинга (процессор, память, задержка запросов и т. д.).
- Резервируйте определенный запас ресурсов для обработки скачков трафика.
