# Транзакция базы данных

## Введение

Узел транзакции базы данных выполняет группу операций с базой данных в одной транзакции. Он подходит для сценариев, где несколько шагов с данными должны либо все завершиться успешно, либо все откатиться — например, создание заказа, списание запаса, запись позиций заказа и обновление статуса.

Сейчас узел транзакции поддерживает только источники данных типа базы данных. Операции с данными из того же источника данных внутри узла автоматически включаются в транзакцию; другие источники данных эту транзакцию не используют.

## Создание узла

В интерфейсе настройки рабочего процесса нажмите кнопку плюса («+») в потоке, чтобы добавить узел «Транзакция базы данных».

![20260610205146](https://static-docs.nocobase.com/20260610205146.png)

После создания формируются две ветви:

- **Выполнение**: основная ветвь, выполняемая внутри транзакции. Если все узлы в этой ветви завершаются успешно, транзакция фиксируется автоматически. Если какой-либо узел завершается с ошибкой или выбрасывает исключение, транзакция откатывается автоматически.
- **После отката**: ветвь, выполняемая после отката. Эта ветвь работает вне транзакции и может использоваться для записи логов, отправки уведомлений или компенсирующей обработки.

![20260610205303](https://static-docs.nocobase.com/20260610205303.png)

## Настройка узла

![20260610205505](https://static-docs.nocobase.com/20260610205505.png)

### Источник данных

Выберите источник данных базы данных, управляемый этой транзакцией. В транзакцию автоматически включаются только узлы операций с данными из того же источника данных.

### Уровень изоляции

Задайте уровень изоляции транзакции. Значение по умолчанию — `READ UNCOMMITTED`. Если бизнесу требуется более строгая согласованность данных, выберите другой уровень изоляции с учётом возможностей базы данных и требований к параллелизму.

### Продолжить рабочий процесс после отката

При включении рабочий процесс после завершения ветви `После отката` продолжается к узлам, следующим за узлом транзакции.

При отключении рабочий процесс после завершения ветви `После отката` останавливается на узле транзакции, и последующие узлы не выполняются.

## Использование

### Ограничения

Ветвь `Выполнение` не поддерживает асинхронные узлы, приостанавливающие рабочий процесс, например узлы ручной обработки и задержки. Транзакция должна быть зафиксирована или откачена в рамках текущего выполнения. Если ветвь `Выполнение` переходит в состояние ожидания, система откатывает транзакцию и помечает рабочий процесс как неудачный.

Ветвь `После отката` выполняется вне транзакции, поэтому на неё не распространяется указанное ограничение. В этой ветви при необходимости можно использовать асинхронные узлы, например отправку запросов, ожидание ручного подтверждения или отложенную обработку.

:::warning Примечание
Транзакции занимают соединения с базой данных до фиксации или отката. Избегайте длительных операций в ветви `Выполнение` и оставляйте там только необходимые чтение, запись и проверки данных.
:::

### Вложенные транзакции

Узлы транзакции могут быть вложенными, но нужно учитывать область источника данных:

- Если внутренняя и внешняя транзакции используют один источник данных, внутренняя транзакция создаётся в области внешней и обрабатывается в соответствии с возможностями базы данных и Sequelize.
- Если внутренняя транзакция использует другой источник данных, она не переиспользует внешнюю транзакцию и создаёт независимую транзакцию для этого источника данных.
- Если рабочий процесс запущен синхронным событием коллекции, сам триггер может уже предоставлять транзакцию верхнего уровня для того же источника данных. Узел транзакции в приоритете переиспользует внешнюю транзакцию того же источника данных и не переиспользует транзакции из других источников данных.

Вложенные транзакции усложняют понимание и устранение неполадок. Как правило, используйте вложенные транзакции только когда действительно нужна локальная граница отката. В остальных случаях предпочтительнее обернуть полный поток обработки данных в один узел транзакции.

### Типичный сценарий

Типичный поток выглядит так:

1. Запросите или создайте связанные данные в ветви `Выполнение`.
2. Продолжите обновление запаса, статуса, позиций и других данных из того же источника данных в ветви `Выполнение`.
3. Если все шаги завершаются успешно, транзакция фиксируется автоматически.
4. Если какой-либо узел завершается с ошибкой или выбрасывает исключение, транзакция откатывается автоматически, и рабочий процесс переходит в ветвь `После отката`.
5. Зафиксируйте причину сбоя, отправьте уведомления или выполните компенсирующую логику в ветви `После отката`.

Если рабочий процесс должен продолжиться после отката, включите «Продолжить рабочий процесс после отката».
